# Htimer

`htimer` — backend-сервис для управления проектами и рабочими процессами команды: пользователи, проекты, этапы, задачи, ежедневные отчёты (daily logs), подписки, платежи и генерация отчётов.

## Бизнес-ценность проекта

`htimer` создаётся как единый операционный контур для командной проектной работы: от планирования и исполнения до контроля результата и отчётности.

### Какую проблему решает

Во многих командах данные о ходе проекта распылены между чатами, таблицами и разрозненными сервисами. Это приводит к:

- потере контекста и ручной сверке статусов;
- слабой прозрачности по загрузке и прогрессу;
- сложностям с управлением доступами и ответственностью;
- задержкам в управленческой отчётности.

### Ценность для бизнеса

- единая точка правды по проектам, задачам, этапам и daily logs;
- быстрее управленческие решения за счёт структурированной и актуальной отчётности;
- снижение операционных затрат на ручной контроль и сбор статусов;
- лучшее качество управляемости через формализованные роли, статусы и бизнес-правила;
- готовая база для масштабирования на несколько команд/проектов без потери прозрачности.

### Целевые пользователи и стейкхолдеры

- руководитель проекта / владелец продукта — контроль сроков, рисков, прогресса;
- администратор/операционный менеджер — управление ролями, составом и процессной дисциплиной;
- исполнители — понятный контур задач, этапов и ежедневной фиксации результатов;
- финансовый контур — подписки и платежные события в едином процессе.

### Продуктовый эффект (что можно измерять)

На уровне KPI проект ориентирован на:

- сокращение времени на подготовку проектной отчётности;
- снижение доли «просроченных без видимой причины» задач;
- уменьшение времени на онбординг участника в проект;
- рост предсказуемости исполнения этапов;
- снижение количества ручных операций в процессах контроля.

### Монетизационная логика

В архитектуре уже учтён фундамент monetization-ready модели:

- подписки на уровне проекта;
- платежный контур через YooKassa;
- обработка статусов оплаты и рефанд-сценариев.

Это позволяет развивать продукт как B2B/B2B2C сервис с подписочной моделью и прозрачным управлением оплатами.

## Статус проекта

Проект находится **на стадии активной разработки**.

- текущая цель — довести функциональность до стабильного MVP;
- часть модулей уже используется в интеграционных тестах и бизнес-потоках;
- API-слой пока частично каркасный (видны заготовки и незаполненные обработчики/регистрации роутов);
- после MVP запланирован целевой рефакторинг архитектуры и домена.

## Важная оговорка по доменной модели

На текущем этапе доменная модель получилась **частично анемичной** (часть поведения и правил живёт в интеракторах/политиках и инфраструктурных оркестрациях). Это осознанный компромисс из-за сроков.

После MVP запланирован рефакторинг:

- усиление инкапсуляции бизнес-инвариантов внутри доменных сущностей и value objects;
- уменьшение «процедурного» кода в интеракторах;
- более строгая граница между политиками доступа, доменными правилами и инфраструктурой;
- унификация модели ошибок и контрактов между слоями.

## Предметная область (что покрывает система)

Сервис ориентирован на сопровождение проектной деятельности:

- пользователи и роли (администратор/исполнитель), управление доступом;
- проекты и состав участников;
- этапы (включая древовидную структуру подэтапов);
- задачи по этапам;
- daily logs по проектам/этапам;
- файлы для daily logs (через объектное хранилище);
- подписки и платежные операции;
- отчёты по проекту/пользователям и их асинхронная генерация через брокер сообщений.

## Архитектурный подход

Проект следует слоистой структуре, близкой к Clean Architecture / Hexagonal:

- `domain/` — сущности, enum-статусы, value objects, базовые доменные решения;
- `application/` — DTO, валидаторы, интеракторы (use cases), контракты и прикладные исключения;
- `infrastructure/` — реализации внешних портов: БД, логирование, auth, MinIO, RabbitMQ, платежный шлюз;
- `handlers/` — адаптеры входа (HTTP/API и consumers для брокера);
- `ioc/` — wiring зависимостей через `dishka`;
- `tests/integration/` — интеграционные тесты инфраструктурных и репозиторных сценариев.

## Модули и ответственность

### `application/*`

По каждому бизнес-направлению (`user`, `project`, `stage`, `task`, `daily_log`, `subscription`, `reports`) выделены:

- `dto.py` — входные/выходные модели use case;
- `validators.py` — синтаксические и часть прикладных проверок;
- `interfaces.py` — порты прикладного слоя;
- `interactors.py` — оркестрация сценариев;
- `exceptions.py` — прикладные ошибки.

### `domain/*`

- `entities.py` — основные сущности и статусы домена;
- `value_objects.py` — value objects;
- `services.py` — доменные сервисы/протоколы (пока минимальный слой).

### `infrastructure/db/*`

- SQLAlchemy-модели с ограничениями (`UniqueConstraint`, `CheckConstraint`, индексы);
- async-engine/session factory;
- репозитории БД (низкоуровневый доступ);
- миграции Alembic.

### `infrastructure/repositories/*`

Адаптер-слой над БД/хранилищем:

- маппинг инфраструктурных исключений в `application.common_exceptions`;
- объединение нескольких источников (пример: `FileRepository` для БД + object storage).

### `handlers/consumers/*`

Асинхронные обработчики сообщений (например, генерация отчёта):

- чтение очереди;
- retry-механика через заголовки сообщения;
- dead-letter сценарий;
- обновление статуса отчёта при исчерпании попыток.

### `handlers/api/v1/*`

HTTP-слой пока частично заготовочный:

- есть базовый каркас `routers` и `exception_handlers`;
- часть обработчиков/схем в работе;
- требуется дальнейшая консолидация импортов, регистраций роутов и контрактов ответов.

## Конфигурация

Конфигурация построена через `pydantic-settings` в `src/config.py`:

- Postgres
- FastAPI
- JWT
- MinIO
- RabbitMQ
- YooKassa
- сервисные блоки (`Clock`, `Logger`, хост/безопасность)

Примечание: в текущем виде присутствуют дефолтные значения, удобные для разработки. Перед production-профилем требуется жёсткая настройка через окружение и аудит секретов.

## Асинхронная обработка и брокер

В проекте реализован отдельный runtime для consumers:

- инициализация топологии RabbitMQ;
- запуск обработчиков в конкурентном режиме;
- retry + dead-letter маршрутизация для ошибок;
- логирование успешных/ошибочных этапов генерации отчётов.

## Хранение и файлы

Для файлов daily logs используется object storage (MinIO):

- создание upload/download/remove ссылок;
- согласование состояния БД и хранилища;
- откаты транзакций в неуспешных ветках.

## Платежи и подписки

Сценарий монетизации включает:

- создание платежа через YooKassa;
- проверку статуса оплаты;
- рефанд;
- применение платежа к подписке проекта.

На текущем этапе важен дополнительный hardening обработки внешних ошибок и edge-cases платёжного API.

## Тестовое покрытие (интеграционный слой)

`src/tests/integration` покрывает:

- инфраструктурные компоненты (clock, auth, logger, text normalizer, payment gateway);
- DB-репозитории по основным агрегатам;
- main repositories (маппинг и контракты);
- тестовые фикстуры DI-контейнера и изолированные транзакционные сценарии.

## Известные ограничения и техдолг

На текущий момент:

- часть API-слоя и OpenAPI-описания ещё не доведена до целостного production-контура;
- требуется выравнивание точек входа и финальная стабилизация CLI/API wiring;
- доменная модель нуждается в деанемизации (см. отдельную оговорку выше).

## План развития после MVP

Приоритетные направления:

1. Рефакторинг доменной модели и инвариантов (устранение анемичности).
2. Полная консолидация HTTP API: роутинг, схемы, ошибки, документация.
3. Усиление observability: структурные логи, метрики, трассировка фоновых задач.
4. Расширение тестов (интеграционные + целевые контрактные/сценарные).
5. Подготовка production-профиля конфигурации и security hardening.
